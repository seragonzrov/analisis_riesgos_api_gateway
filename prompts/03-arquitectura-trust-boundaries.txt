# Prompt 03: Arquitectura y Trust Boundaries

**Fecha:** 24/05/2026
**Herramienta:** Claude Sonnet 4.6 (Anthropic)
**Propósito:** Documentar arquitectura del sistema y definir las zonas de confianza

---

## Prompt Utilizado

```
Tengo el siguiente sistema de API Gateway para microservicios.
Usá el contexto del 00-memory-bank.md como referencia.

Tecnologías del sistema:
- API Gateway: Kong / AWS API Gateway con OAuth2/JWT y rate limiting
- WAF: ModSecurity / AWS WAF como capa perimetral
- Service Mesh: Istio con mTLS y Envoy sidecars
- Orquestación: Kubernetes cluster
- Microservicios: Users, Inventory, Orders (containers Docker)
- Datos: PostgreSQL, RabbitMQ, Elasticsearch
- Cache/Sesiones: Redis
- Logging: ELK Stack (Elasticsearch, Logstash, Kibana)

Por favor generá:

1. DIAGRAMA DE ARQUITECTURA en ASCII:
   Las 4 zonas del sistema con los componentes dentro de cada una
   y flechas indicando protocolos entre zonas.

2. FLUJO DE DATOS paso a paso:
   - Cómo fluye una request desde el cliente hasta la base de datos
   - Qué hace cada componente en ese flujo
   - Flujos adicionales: async messaging, logging, métricas, security events

3. TABLA DE ACTORES DEL SISTEMA:
   Columnas: Actor | Descripción | Privilegios

4. DEFINICIÓN DE TRUST BOUNDARIES:
   Para cada zona (TB1 a TB4), especificá:
   - Nombre y nivel de confianza (%)
   - Protocolo obligatorio para cruzar el boundary
   - Mecanismo de validación
   - Qué NO pasa automáticamente sin validación explícita

Regla clave: no existe confianza transitiva entre boundaries.
Formato Markdown con diagramas ASCII.
```

---

## Response Summary

La IA generó:
- Diagrama ASCII de la arquitectura en 4 capas con componentes y protocolos
- Flujo de requests en 5 pasos principales:
  1. Cliente → WAF (HTTPS, filtrado de ataques, rate limiting por IP)
  2. WAF → API Gateway (JWT/OAuth2, rate limiting por usuario, logging)
  3. API Gateway → Service Mesh → Microservicio (mTLS, routing, circuit breaking)
  4. Microservicio → Base de Datos (connection pool, prepared statements)
  5. Microservicio → Response → Cliente (logging, compresión, caching)
- 4 flujos adicionales: async messaging (RabbitMQ), logging (Fluentd → Elasticsearch),
  métricas (Istio → Prometheus → Grafana) y security events (Falco → PagerDuty)
- Tabla de 9 actores del sistema con privilegios
- 4 trust boundaries definidos:
  - TB1: Internet ↔ DMZ (0% → 30%, HTTPS + WAF + JWT)
  - TB2: DMZ ↔ Service Mesh (30% → 70%, mTLS + certificados Istio)
  - TB3: Service Mesh ↔ Data Layer (70% → 90%, TLS + credentials rotation)
  - TB4: Usuarios ↔ K8s Control Plane (variable, kubectl + RBAC + audit logging)
