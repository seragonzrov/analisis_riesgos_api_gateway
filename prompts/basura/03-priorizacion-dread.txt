## CONTEXTO
Eres un experto en seguridad ofensiva y defensiva especializado en arquitecturas de microservicios.
Estás realizando un threat modeling formal para un sistema compuesto por:
- API Gateway (Kong / AWS API Gateway) con autenticación OAuth2/JWT
- Service Mesh con Istio y mTLS
- Kubernetes cluster con 3 microservicios (Users, Inventory, Orders)
- PostgreSQL, RabbitMQ, Elasticsearch, Redis
- ELK Stack para logging centralizado

## TAREA
Tomá la siguiente lista de amenazas STRIDE identificadas y aplicá la metodología DREAD para calcular el score de riesgo de cada una.

Para cada amenaza, evaluá las cinco dimensiones en una escala de 1 a 10:

- **Damage (D):** ¿Qué tan grave es el daño si se explota exitosamente?
  1-3 = daño menor (datos no críticos, sin impacto en servicio)
  4-6 = daño moderado (acceso parcial, interrupción temporal)
  7-9 = daño severo (acceso masivo, interrupción extendida)
  10  = daño catastrófico (compromiso total del sistema)

- **Reproducibility (R):** ¿Qué tan fácil es reproducir el ataque?
  1-3 = muy difícil (requiere condiciones muy específicas)
  4-6 = moderado (posible con esfuerzo)
  7-9 = fácil (herramientas disponibles, condiciones comunes)
  10  = siempre funciona (100% reproducible)

- **Exploitability (E):** ¿Qué habilidades técnicas se necesitan?
  1-3 = requiere expertise muy avanzado y recursos significativos
  4-6 = requiere conocimiento técnico moderado
  7-9 = posible con herramientas automatizadas (Burp, sqlmap, Metasploit)
  10  = no requiere habilidades, cualquier persona puede hacerlo

- **Affected Users (A):** ¿Cuántos usuarios se ven impactados?
  1-3 = usuario individual o grupo muy pequeño
  4-6 = grupo de usuarios (10-100)
  7-9 = muchos usuarios (100-10.000)
  10  = todos los usuarios del sistema

- **Discoverability (D):** ¿Qué tan fácil es descubrir la vulnerabilidad?
  1-3 = muy difícil (requiere acceso interno o conocimiento específico)
  4-6 = moderado (requiere fuzzing o análisis activo)
  7-9 = fácil (scanners automáticos la detectan, información pública)
  10  = trivial (visible en la UI o documentación pública)

**Score total = D + R + E + A + D (máximo 50)**

Escala de severidad:
- 40-50: CRÍTICO → Remediar de inmediato (sprint actual)
- 30-39: ALTO    → Remediar ASAP (próximo sprint)
- 20-29: MEDIO   → Remediar en 2-3 sprints
- 10-19: BAJO    → Monitorear y evaluar

## CONSIDERACIONES ADICIONALES
- Tener en cuenta el contexto específico de la arquitectura (Kubernetes, Istio mTLS) al evaluar exploitability
- Para amenazas de Denial of Service, Affected Users = 10 si afecta la disponibilidad total del servicio
- Penalizar con Discoverability alta (8-10) cualquier vulnerabilidad detectable por scanners automáticos como OWASP ZAP, Burp Suite, sqlmap o Nikto
- Si una amenaza tiene mitigación parcial ya implementada (ej: WAF presente), reflejarlo bajando el score de Exploitability en 1-2 puntos y justificarlo
