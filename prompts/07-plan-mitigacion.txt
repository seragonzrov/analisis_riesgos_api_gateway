# Prompt 07: Plan de Mitigación

**Fecha:** 24/05/2026
**Herramienta:** Claude Sonnet 4.6 (Anthropic)
**Propósito:** Diseñar controles de seguridad priorizados para mitigar las amenazas identificadas

---

## Prompt Utilizado

```
Eres un arquitecto de seguridad con experiencia en sistemas cloud-native,
Kubernetes y API security. Contexto del sistema en 00-memory-bank.md.

Tenés el análisis completo:
- 27 amenazas STRIDE en docs/04-analisis-stride.md
- Scores DREAD en docs/05-priorizacion-dread.md
- Kill chain en docs/06-mapa-attack.md

Las 4 amenazas CRÍTICAS (DREAD ≥ 40) son:
- TH18: API Flooding/DDoS — 46/50
- TH25: IDOR — 42/50
- TH06: SQL Injection — 41/50
- TH14: PII Exposure in Logs — 40/50

El kill chain identificado tiene 11 fases (ver docs/06-mapa-attack.md):
Initial Access → Execution → Persistence → Privilege Escalation →
Defense Evasion → Credential Access → Discovery → Lateral Movement →
Collection → Exfiltration → Impact

Diseñá un plan de mitigación completo ejecutable en sprints de 2 semanas.

PARTE 1 — TABLA DE CONTROLES:
Una tabla con todos los controles con columnas:
ID | Amenazas que mitiga | Control de Seguridad | Prioridad | Estado

Prioridades:
- P0 (Crítico): implementación inmediata, Sprint 1-2
- P1 (Alto): próximo ciclo, Sprint 3-4
- P2 (Medio): planificado, Sprint 5-6

PARTE 2 — CONTROLES POR CATEGORÍA:
Organizá los controles en tres categorías:
- Controles PREVENTIVOS (evitan que ocurra el ataque)
- Controles DETECTIVOS (detectan cuando ocurre el ataque)
- Controles CORRECTIVOS (responden y recuperan después del ataque)

PARTE 3 — DETALLE DE TODOS LOS CONTROLES (C01-C15):
Para cada control, generá una ficha completa con:
- Amenazas que mitiga (todas, no solo la principal)
- Prioridad, timeline y responsable sugerido
- Implementación técnica paso a paso
- Snippets de configuración reales:
  YAML de Kubernetes, código JavaScript/TypeScript,
  configuración de Fluentd, comandos de Vault,
  reglas de Falco, comandos bash, etc.
- Criterio de aceptación con checklist
- Esfuerzo estimado en horas y costo (open source / licencia / cloud)

Los controles P0 deben poder implementarse en menos de 16 horas de ingeniería.
Preferir herramientas open source ya presentes en el stack.
```
